# 逾期处理链路分析

## 一、代码结构概览

逾期处理涉及三个核心层级，关键文件如下：

| 层级 | 文件 | 职责 |
|------|------|------|
| Scheduler | `OverdueNotificationScheduler.java` | 定时调度：逾期检查、通知发送、记录更新、结清处理 |
| Service | `BorrowRequestService.java` / `ScanService.java` | 归还业务：设置 actualReturnDate、即时结清逾期记录 |
| Service | `OverdueRecordService.java` | 逾期记录 CRUD：手动创建/更新/结清逾期记录 |
| Entity | `BorrowRequest.java` | 借用申请实体，含 `overdueNotified`、`dueSoonNotified` 标记 |
| Entity | `OverdueRecord.java` | 逾期记录实体，含 `overdueDays`、`resolved` 状态 |

---

## 二、定时任务执行流程

调度器每天 **9:00** 执行，入口方法 `checkBorrows()` 依次调用三个子任务：

```
checkBorrows()
  ├── 1. checkDueSoonBorrows(today)       — 即将到期提醒
  ├── 2. checkOverdueBorrows(today)        — 逾期检查 + 通知 + 记录更新
  └── 3. resolveReturnedOverdueRecords()    — 结清已归还的逾期记录
```

### 2.1 checkOverdueBorrows 核心逻辑

这是逾期处理的主入口，伪代码如下：

```
查询条件: status = APPROVED AND expectedReturnDate < today AND actualReturnDate IS NULL
对每条逾期借用记录:
  ├── 计算 overdueDays = today - expectedReturnDate
  ├── if (!borrowRequest.isOverdueNotified()):    ← 仅首次逾期时进入
  │     ├── 发送 BORROW_OVERDUE 通知给借用者
  │     ├── 发送 BORROW_OVERDUE 通知给工具所有者
  │     └── 设置 borrowRequest.setOverdueNotified(true)
  └── createOrUpdateOverdueRecord(borrowRequest, today, overdueDays)  ← 每次都执行
        ├── 若已有记录 → 更新 overdueDays
        └── 若无记录   → 新建 OverdueRecord
```

**关键差异一目了然：**
- **逾期通知**：受 `overdueNotified` 标记守卫，只在首次逾期时发送一次
- **逾期记录**：不受任何标记守卫，每次调度都会执行 `createOrUpdateOverdueRecord`

## 三、通知与记录更新策略差异的设计意图

### 3.1 通知只发一次——避免通知轰炸

逾期通知使用 `BorrowRequest.overdueNotified` 布尔标记做幂等守卫：

```java
// OverdueNotificationScheduler.java 第117行
if (!borrowRequest.isOverdueNotified()) {
    // 发送通知...
    borrowRequest.setOverdueNotified(true);
}
```

设计意图是**用户体验优先**：逾期本身就是异常状态，如果每天 9 点都推送一条"您已逾期 N 天"的通知，对借用者而言是重复骚扰，对工具所有者而言也是噪音。一次通知足以让双方知情，后续催还应通过其他渠道（如人工联系）处理。

同理，`dueSoonNotified` 也只发一次即将到期通知，避免在到期前 1~3 天内每天重复提醒。

### 3.2 逾期天数每天更新——保持事实准确性

逾期天数是一个**随时间递增的事实指标**，不是一次性的状态标记：

```java
// OverdueNotificationScheduler.java 第158-160行
if (existingOpt.isPresent()) {
    record = existingOpt.get();
    record.setOverdueDays(overdueDays);  // 每天刷新
}
```

设计意图是**数据准确性优先**：`OverdueRecord.overdueDays` 是逾期严重程度的量化指标，用于：
- 管理后台的逾期统计面板（`StatsService` 读取 `countByResolved`）
- 逾期记录列表的排序和展示（`OverdueRecordService.getAllOverdueRecords`）
- 管理员判断是否需要采取进一步措施

如果逾期天数不每天更新，管理员看到的逾期记录将永远停留在创建时的天数，无法反映真实逾期时长。

### 3.3 两种策略的对比总结

| 维度 | 逾期通知 | 逾期记录（overdueDays） |
|------|---------|----------------------|
| 更新频率 | 仅首次 | 每天 |
| 幂等守卫 | `overdueNotified` 标记 | 无（始终执行） |
| 设计目标 | 避免重复骚扰 | 保持数据时效性 |
| 数据性质 | 一次性事件 | 持续递增的度量值 |
| 用户感知 | 推送型（主动通知） | 拉取型（查询时展示） |

## 四、归还时序分析

归还操作有两个入口：
1. **常规归还**：`BorrowRequestService.updateBorrowStatus()` → `case RETURNED`
2. **扫码归还**：`ScanService.returnTools()`

两者在归还时都会：
- 设置 `borrowRequest.setActualReturnDate(LocalDate.now())`
- 即时结清逾期记录（`record.setResolved(true)`）

下面分析归还动作发生在调度任务执行前/后的各种场景。

### 4.1 场景一：归还发生在调度任务当天执行之前

```
时间线:
  Day N-1 23:59  借用已逾期（expectedReturnDate = Day N-2）
  Day N   08:30  用户归还工具 → actualReturnDate = Day N, overdueRecord.resolved = true
  Day N   09:00  定时任务执行
```

**调度任务行为：**
- `checkOverdueBorrows()` 的查询条件是 `actualReturnDate IS NULL`，由于归还已设置该字段，**该借用记录不会被查到**
- 不会发送逾期通知 ✅（已归还，无需通知）
- 不会更新逾期记录 ✅（记录已 resolved）

**潜在问题：**
- ❌ **首次逾期通知可能漏发**：如果 Day N 是逾期的第一天（expectedReturnDate = Day N-1），且用户在 9:00 前归还，则逾期通知永远不会发出。用户确实存在逾期行为，但系统没有任何通知记录。
- ❌ **逾期记录可能缺失**：同样，如果这是逾期第一天且用户在调度前归还，`OverdueRecord` 表中不会有任何记录。从审计角度看，这次逾期事件"消失"了。
- ⚠️ **影响程度评估**：用户已主动归还，问题已实质性解决。对于"友好提醒"型系统可以接受；对于合规审计型系统则存在数据缺失。

### 4.2 场景二：归还发生在调度任务当天执行之后

```
时间线:
  Day N   09:00  定时任务执行
  Day N   15:00  用户归还工具 → actualReturnDate = Day N, overdueRecord.resolved = true
```

**调度任务行为：**
- `checkOverdueBorrows()` 查到该记录（actualReturnDate 仍为 null）
- 如果是首次逾期：发送通知 ✅，创建逾期记录 ✅
- 如果非首次逾期：更新 overdueDays ✅
- `resolveReturnedOverdueRecords()` 不会处理（此时 actualReturnDate 仍为 null）

**归还后行为：**
- 归还处理器即时设置 `resolved = true` ✅
- 逾期通知已正确发送 ✅
- 逾期天数已正确更新 ✅

**潜在问题：**
- ⚠️ **逾期天数快照偏差**：`OverdueRecord.overdueDays` 记录的是调度执行时刻的值（如 Day N 9:00 计算的天数），而非归还时刻的值。由于两者在同一天，`overdueDays` 值相同，**无实际偏差**。

### 4.3 场景三：归还发生在调度任务次日执行之前（跨天未调度）

```
时间线:
  Day N   09:00  定时任务执行，overdueDays = 2
  Day N+1 08:30  用户归还工具 → actualReturnDate = Day N+1, overdueRecord.resolved = true
  Day N+1 09:00  定时任务执行
```

**调度任务行为（Day N+1 9:00）：**
- 查询条件 `actualReturnDate IS NULL`，该记录已被归还，**不会被查到**
- 不会重复更新逾期记录 ✅

**潜在问题：**
- ⚠️ **逾期天数少算 1 天**：`OverdueRecord.overdueDays` 停留在 Day N 调度时计算的值（如 2 天），但实际归还日是 Day N+1（实际逾期 3 天）。归还处理器只做 `resolved = true`，**不会更新 overdueDays**。
- 不过，`BorrowRequestService.populateOverdueFields()` 在查询时会实时计算逾期天数，所以前端展示的逾期天数是准确的。问题仅存在于 `OverdueRecord` 表的持久化数据中。

### 4.4 场景四：并发归还与调度

```
时间线:
  Day N 09:00:00.000  调度事务开始，读取 BorrowRequest（actualReturnDate = null）
  Day N 09:00:00.100  归还事务开始，设置 actualReturnDate，resolved = true
  Day N 09:00:00.200  调度事务继续，基于已读取的数据创建/更新逾期记录
```

**潜在问题：**
- ⚠️ **脏读/幻读风险**：调度事务基于快照读取了 `actualReturnDate = null` 的数据，但归还事务已修改了该记录。由于没有显式加锁（如 `SELECT ... FOR UPDATE`），调度事务可能基于过期数据操作。
- **实际影响有限**：
  - 如果归还事务先提交，调度事务的 `createOrUpdateOverdueRecord` 会写入一条 `resolved = false` 的记录，但 `resolveReturnedOverdueRecords()` 在同一调度周期内会将其标记为 `resolved = true`
  - 如果调度事务先提交，归还事务的即时结清逻辑会正确处理
- **最坏情况**：逾期通知被重复发送给用户（调度事务发送了通知，但归还事务也触发了 `BORROW_RETURNED` 通知）。两种通知类型不同，不会造成用户困惑，但逾期通知本身是多余的。

## 五、双重逾期天数来源的一致性问题

系统中存在两套逾期天数计算逻辑，可能产生不一致：

### 5.1 OverdueRecord.overdueDays（持久化）

由调度器在每天 9:00 计算：

```java
// OverdueNotificationScheduler.java 第115行
int overdueDays = (int) ChronoUnit.DAYS.between(borrowRequest.getExpectedReturnDate(), today);
```

**特点**：只在调度时刻更新，归还后不再变化（历史快照）。

### 5.2 BorrowRequestResponse.overdueDays（实时计算）

由 `populateOverdueFields()` 在每次查询时实时计算：

```java
// BorrowRequestService.java 第346行
response.setOverdueDays((int) ChronoUnit.DAYS.between(borrowRequest.getExpectedReturnDate(), today));
```

**特点**：查询时实时计算，始终反映当前时刻的逾期天数。

### 5.3 不一致场景

| 时刻 | OverdueRecord.overdueDays | BorrowRequestResponse.overdueDays | 是否一致 |
|------|--------------------------|----------------------------------|---------|
| Day N 8:00（调度前） | 昨天的值（如 2） | 3 | ❌ |
| Day N 9:30（调度后） | 3 | 3 | ✅ |
| Day N+1 8:30（归还后，调度前） | 3 | 0（已归还） | ❌ |
| Day N+1 10:00（归还后，调度后） | 3（已 resolved，不再更新） | 0（已归还） | ❌ |

**注意**：两者服务于不同场景——`OverdueRecord` 是历史审计记录，`BorrowRequestResponse` 是当前状态展示，语义不同，不一致是预期行为。但如果前端混用两个数据源展示"逾期天数"，则会出现显示不一致。

## 六、问题汇总与风险评估

### 6.1 已识别问题

| 编号 | 问题 | 严重程度 | 触发条件 | 影响 |
|------|------|---------|---------|------|
| P1 | 首日逾期通知漏发 | 中 | 用户在逾期首日 9:00 前归还 | 逾期事件无通知记录 |
| P2 | 首日逾期记录缺失 | 中 | 同上 | OverdueRecord 表无该次逾期记录 |
| P3 | 逾期天数少算 | 低 | 归还发生在两次调度之间 | OverdueRecord.overdueDays 比实际少 1 天 |
| P4 | 调度前数据不一致窗口 | 低 | 调度执行前的查询 | 两个数据源逾期天数不一致 |
| P5 | 并发竞态条件 | 极低 | 调度与归还同时执行 | 可能多发一次逾期通知 |

### 6.2 现有安全机制

系统已内置以下安全机制，可缓解部分问题：

1. **即时结清**：`BorrowRequestService` 和 `ScanService` 在归还时立即 `resolved = true`，无需等待下次调度
2. **兜底结清**：`resolveReturnedOverdueRecords()` 在每次调度末尾执行，确保遗漏的记录被处理
3. **实时计算**：`populateOverdueFields()` 在查询时实时计算逾期状态，前端展示始终准确
4. **查询条件守卫**：`findOverdueBorrows` 同时检查 `status = APPROVED` 和 `actualReturnDate IS NULL`，双重过滤已归还记录

## 七、核心链路时序图

### 7.1 正常逾期流程（无归还）

```
Day N-1          Day N 09:00                    Day N+1 09:00
  │                 │                              │
  │  expectedReturn  │  checkOverdueBorrows()       │  checkOverdueBorrows()
  │  Date 到期       │  ├─ overdueNotified=false     │  ├─ overdueNotified=true
  │                 │  ├─ 发送 BORROW_OVERDUE ✉️     │  ├─ 跳过通知（已发过）
  │                 │  ├─ setOverdueNotified(true)   │  └─ update overdueDays=2
  │                 │  └─ create OverdueRecord       │
  │                 │     overdueDays=1              │
```

### 7.2 逾期首日调度前归还

```
Day N-1          Day N 08:30              Day N 09:00
  │                 │                        │
  │  expectedReturn  │  用户归还               │  checkOverdueBorrows()
  │  Date 到期       │  ├─ actualReturnDate=today  │  ├─ 查询条件: actualReturnDate IS NULL
  │                 │  ├─ resolved=true ❌无记录可结清  │  │  → 该记录已被排除
  │                 │  └─ 发送 BORROW_RETURNED ✉️    │  └─ 无操作
  │                 │                        │
  │                 │  ⚠️ 逾期通知未发出            │
  │                 │  ⚠️ OverdueRecord 未创建       │
```

### 7.3 逾期后调度执行再归还

```
Day N-1          Day N 09:00                 Day N 15:00
  │                 │                            │
  │  expectedReturn  │  checkOverdueBorrows()       │  用户归还
  │  Date 到期       │  ├─ 发送 BORROW_OVERDUE ✉️     │  ├─ actualReturnDate=today
  │                 │  ├─ create OverdueRecord     │  ├─ resolved=true ✅
  │                 │  │  overdueDays=1            │  └─ 发送 BORROW_RETURNED ✉️
  │                 │  └─ setOverdueNotified(true)  │
  │                 │                            │
  │                 │  ✅ 通知已发送                  │
  │                 │  ✅ 记录已创建并结清              │
```

## 八、结论

现有设计在**用户体验**和**数据准确性**之间做了合理的取舍：

- **通知只发一次**是正确的选择，避免了逾期期间的重复骚扰
- **逾期天数每天更新**也是必要的，保证了管理端数据的时效性
- **即时结清 + 兜底结清**的双保险机制有效防止了记录状态不一致

主要风险集中在**逾期首日调度前归还**这一边界场景：逾期事件可能从系统中"消失"。对于当前"友好提醒"型系统，这是可接受的取舍；若未来需要严格的逾期审计追踪，则需要补充事件驱动机制（如在归还时回溯检查并补录逾期记录）。
