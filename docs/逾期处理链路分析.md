## 3. 调度任务主流程

调度任务定义在 [OverdueNotificationScheduler.java](file:///d:/charles/program/ai/apps/02.work%20session/session-gsb0617/source%20code/app-00012/app-00012-saturn/src/main/java/com/toolshare/scheduler/OverdueNotificationScheduler.java) 中，每天早上 9:00 执行一次（cron: `0 0 9 * * ?`）。

执行顺序如下：

1. **checkDueSoonBorrows(today)** — 检查即将到期的借用（提前 3 天）
2. **checkOverdueBorrows(today)** — 检查已逾期的借用
3. **resolveReturnedOverdueRecords()** — 处理已归还的逾期记录

整个方法标注了 `@Transactional`，在一个事务中完成所有操作。

## 4. 逾期通知 vs 逾期记录：策略差异分析

### 4.1 逾期通知策略：一次性发送

**核心逻辑**位于 [OverdueNotificationScheduler.java#L104-L152](file:///d:/charles/program/ai/apps/02.work%20session/session-gsb0617/source%20code/app-00012/app-00012-saturn/src/main/java/com/toolshare/scheduler/OverdueNotificationScheduler.java#L104-L152)

```java
if (!borrowRequest.isOverdueNotified()) {
    // 发送通知给借用者和所有者...
    borrowRequest.setOverdueNotified(true);
    toUpdate.add(borrowRequest);

```

**关键特征：**
- 通知由 `BorrowRequest.overdueNotified` 字段控制，默认为 `false`
- 第一次检测到逾期时发送通知，随后将标志位置为 `true`
- 之后即使逾期天数继续增加，也不会再发送通知
- 同样的模式也适用于 "即将到期" 通知（`dueSoonNotified` 字段）

### 4.2 逾期记录策略：每日更新

**核心逻辑**位于 [OverdueNotificationScheduler.java#L154-L172](file:///d:/charles/program/ai/apps/02.work%20session/session-gsb0617/source%20code/app-00012/app-00012-saturn/src/main/java/com/toolshare/scheduler/OverdueNotificationScheduler.java#L154-L172)

```java
private void createOrUpdateOverdueRecord(BorrowRequest borrowRequest, LocalDate today, int overdueDays) {
    Optional<OverdueRecord> existingOpt = overdueRecordRepository.findByBorrowRequestId(borrowRequest.getId());
    OverdueRecord record;

    if (existingOpt.isPresent()) {
        record = existingOpt.get();
        record.setOverdueDays(overdueDays);  // 每天更新逾期天数
    } else {
        // 创建新记录...
    }
    overdueRecordRepository.save(record);
}
```

**关键特征：**
- 逾期记录存储在独立的 `OverdueRecord` 表中，通过 `borrowRequestId` 关联
- 每次调度任务执行时，只要借用仍处于逾期状态，就会更新 `overdueDays` 字段
- 逾期记录有独立的 `resolved` 标志位，表示是否已解决（即已归还）

### 4.3 设计意图解读

两种策略的差异源于它们服务于不同的业务目标：

| 维度 | 逾期通知 | 逾期记录 |
|------|---------|---------|
| **业务目标** | 提醒用户，避免消息轰炸 | 准确记录逾期状态，用于统计和信用评估 |
| **更新频率** | 一次（首次逾期时） | 每日更新 |
| **数据用途** | 用户体验（通知中心） | 运营统计、信用记录、逾期追溯 |
| **控制字段** | `BorrowRequest.overdueNotified` | `OverdueRecord.overdueDays` / `resolved` |
| **设计考量** | 避免每天发通知骚扰用户 | 需要精确的逾期天数用于统计和展示 |

**设计合理性：**
- **通知一次性**：合理。每天发逾期提醒会造成信息骚扰，用户体验差。通常一次通知即可，后续用户可以在应用内查看逾期状态。
- **记录每日更新**：合理。逾期天数是重要的业务指标，需要准确保留，用于后续的信用评估、统计报表等场景。

## 6. 时序一致性问题分析

调度任务每天 9:00 执行一次，而归还操作可以在任意时刻发生。以下分析几种典型的时序场景。

### 6.1 场景一：归还发生在调度任务执行之前

**时间线：**
```
T-1日 23:59  — 借用仍逾期，未归还
T日   08:30  — 用户归还工具（操作发生在调度前）
T日   09:00  — 调度任务执行
```

**调度任务各步骤行为：**

1. **checkOverdueBorrows()**：
   - 查询条件：`status = APPROVED AND expectedReturnDate < today AND actualReturnDate IS NULL`
   - 由于 `actualReturnDate` 已在 08:30 被设置，该借用不会被查询出来
   - **结果**：不会更新逾期天数，也不会发送通知（即使之前未发过）

2. **resolveReturnedOverdueRecords()**：
   - 查询所有 `resolved = false` 的逾期记录
   - 检查对应的 `BorrowRequest.actualReturnDate` 是否非空
   - 由于已归还，会将逾期记录标记为 `resolved = true`
   - **结果**：逾期记录被正确标记为已解决

**结论：** 无问题。归还后调度任务不会重复处理，逾期记录会被正确标记为已解决。

### 6.2 场景二：归还发生在调度任务执行之后

**时间线：**
```
T-1日 23:59  — 借用仍逾期，未归还
T日   09:00  — 调度任务执行
T日   10:00  — 用户归还工具（操作发生在调度后）
```

**调度任务各步骤行为：**

1. **checkOverdueBorrows()**：
   - 该借用会被查询出来（`actualReturnDate` 仍为 null）
   - 若 `overdueNotified = false`，会发送逾期通知
   - 更新逾期天数（`overdueDays` 增加 1 天）

2. **resolveReturnedOverdueRecords()**：
   - 由于尚未归还，逾期记录保持 `resolved = false`

**10:00 归还时的行为：**
- 归还操作会将逾期记录标记为 `resolved = true`
- `BorrowRequest.actualReturnDate` 被设置为 T 日

**潜在问题：**
- **逾期天数准确性**：调度任务在 09:00 将逾期天数更新为 N 天，但用户在当天 10:00 就归还了。从业务角度看，用户实际逾期了 N 天（因为预期归还日是 T-N 日，T 日归还，确实逾期了 N 天），所以这个计算是准确的。
- **通知发送时机**：用户在当天就归还了，但仍然收到了逾期通知。从用户体验角度可能略显严苛，但从业务逻辑角度是正确的——确实发生过逾期。

**结论：** 逻辑正确，无一致性问题。当天归还仍然算逾期（预期归还日已过），逾期天数计算准确。

### 6.3 场景三：首次逾期日的归还（临界场景）

**时间线：**
```
预期归还日 = T-1日
T日 08:30  — 用户归还（首次逾期日的早上，调度前）
T日 09:00  — 调度任务执行
```

**分析：**
- `expectedReturnDate = T-1`，`actualReturnDate = T`
- 从日期计算上，确实逾期了 1 天（T - (T-1) = 1）
- 但由于归还发生在调度之前，`checkOverdueBorrows()` 不会查询到该记录
- **结果**：逾期记录不会被创建，逾期通知也不会发送

**这是一个设计选择问题：**
- 如果认为"只要逾期了哪怕一分钟也应该有记录"，那么这是一个缺陷
- 如果认为"当天早上就还了，情有可原，不必记录"，那么这是可以接受的

当前实现采用的是后者的策略——是否创建逾期记录取决于调度执行时的状态，而不是客观的逾期事实。

### 6.4 场景四：逾期通知漏发风险

**问题：** 什么情况下逾期通知会漏发？

**答案：** 只有一种情况——借用从"即将到期"直接跳到"已归还"，中间没有经过调度任务的逾期检测。

**时间线示例：**
```
T-3日  — 进入"即将到期"范围，发送即将到期通知
T-1日  — 预期归还日
T日 08:30 — 用户归还（调度之前）
T日 09:00 — 调度任务执行，发现已归还，不发逾期通知
```

**分析：**
- 用户收到了"即将到期"通知，但没有收到"已逾期"通知
- 这是合理的——用户确实在逾期第一天就归还了，没有必要再发逾期通知
- 从用户体验角度，这反而是好的设计

**真正的漏发风险：**
- 如果调度任务因服务器故障、部署停机等原因错过执行，那当天的逾期通知和逾期记录更新都会漏掉
- 错过后第二天再执行时：
  - 逾期通知：会补发（因为 `overdueNotified` 还是 false）
  - 逾期记录：会创建/更新，但 `overdueDays` 是基于当前日期计算的，所以仍然准确
- **结论**：通知可能延迟但不会漏发；逾期天数始终准确。

### 6.5 场景五：状态一致性分析

**问题：** 会不会出现 "BorrowRequest 已归还，但 OverdueRecord 仍未解决" 的情况？

**答案：** 不会，原因有二：

1. **归还操作即时处理**：两条归还路径（手动归还和扫码归还）都会主动检查并更新逾期记录的 `resolved` 状态（见 5.1 和 5.2 节）
2. **调度任务兜底处理**：`resolveReturnedOverdueRecords()` 作为兜底机制，每天都会扫描未解决的逾期记录，发现已归还的就标记为已解决

**问题：** 会不会出现 "OverdueRecord 已解决，但 BorrowRequest 还没归还" 的情况？

**答案：** 不会。`resolved` 只会在以下两种情况下被设置为 true：
1. 归还操作时（`actualReturnDate` 同时被设置）
2. 调度任务发现 `actualReturnDate != null` 时

两个条件都保证了 BorrowRequest 一定已经归还。

### 6.6 场景六：逾期天数的一致性

**问题：** `BorrowRequest` 响应中的逾期天数和 `OverdueRecord` 中的逾期天数是否一致？

**答案：** 不一定，存在两处计算逾期天数的逻辑：

1. **API 响应实时计算**（[BorrowRequestService.java#L339-L358](file:///d:/charles/program/ai/apps/02.work%20session/session-gsb0617/source%20code/app-00012/app-00012-saturn/src/main/java/com/toolshare/service/BorrowRequestService.java#L339-L358)）：
   ```java
   response.setOverdueDays((int) ChronoUnit.DAYS.between(borrowRequest.getExpectedReturnDate(), today));
   ```
   每次查询时基于当前日期实时计算。

2. **OverdueRecord 存储的逾期天数**：
   - 由调度任务每天更新一次
   - 在调度任务执行前，可能比实际值少 1 天

**差异举例：**
- 假设调度在 09:00 执行
- 用户在 10:00 查询借用详情，API 返回的逾期天数是实时计算的（准确值）
- 但 OverdueRecord 表中存储的还是昨天更新的值（少 1 天）
- 要等到第二天 09:00 调度执行后，两者才一致

**影响分析：**
- 对用户界面：无影响，因为界面显示的是 API 实时计算的值
- 对统计查询：如果统计基于 OverdueRecord 表，那么在调度执行前查询到的逾期天数可能偏少
- **结论：** 这是一个"最终一致性"设计，数据在每天调度后达到一致。

## 7. 调度任务内部的执行顺序问题

调度任务的三个步骤按固定顺序执行：

```
1. checkDueSoonBorrows()  → 检查即将到期
2. checkOverdueBorrows()  → 检查已逾期
3. resolveReturnedOverdueRecords() → 处理已归还
```

**潜在问题：** 如果一个借用在同一天内既满足"即将到期"又满足"已逾期"条件（理论上不可能，因为即将到期是预期归还日之前，已逾期是预期归还日之后，两者没有交集），所以顺序不影响。

**另一个角度：** 先处理逾期，再处理已归还的逾期记录，顺序是合理的。如果反过来（先标记已解决，再处理逾期），那当天归还的借用就不会被更新逾期天数了——但这其实也没问题，因为已经归还了。

当前顺序保证了：即使当天归还，只要调度时还没归还，逾期天数还是会更新到当天。这是更"严格"的策略。

## 8. 总结

### 8.1 设计意图总结

| 设计选择 | 意图 |
|---------|------|
| 逾期通知只发一次 | 避免消息轰炸，提升用户体验 |
| 逾期记录每日更新 | 保留准确的业务数据，用于统计和信用评估 |
| 归还操作即时更新逾期记录 | 保证用户操作后数据立即可见 |
| 调度任务兜底处理已归还记录 | 容错机制，防止遗漏 |
| API 响应实时计算逾期天数 | 保证用户界面始终显示准确数据 |

### 8.2 一致性问题总结

| 问题 | 是否存在 | 影响程度 | 说明 |
|------|---------|---------|------|
| 逾期通知漏发 | 基本不存在 | 低 | 除非调度任务完全失败，但失败后下次会补发 |
| 逾期天数不准 | 短暂存在 | 低 | OverdueRecord 表中的数据有最多 1 天的延迟，API 实时计算无此问题 |
| 记录状态不一致 | 不存在 | - | 双向校验保证状态一致（归还时即时更新 + 调度兜底） |
| 临界日归还不记逾期 | 是设计选择 | 中 | 逾期第一天调度前归还，不会创建逾期记录 |

### 8.3 观察与建议（仅分析，不修改）

1. **最终一致性模型**：系统采用的是"调度任务最终一致 + API 实时计算"的混合模式，用户界面无感知，但后台统计数据有最多 1 天的延迟。

2. **逾期记录创建时机**：逾期记录不是在逾期发生的那一刻创建的，而是在调度任务执行时创建的。这意味着如果借用在逾期第一天调度前就归还了，就不会有逾期记录。这是一个业务决策点。

3. **调度任务的事务边界**：整个 `checkBorrows()` 方法在一个事务中。如果处理到一半失败，所有更改都会回滚，包括已发送的通知（通知也是数据库记录，会回滚）。这保证了原子性，但如果数据量很大，可能导致长事务问题。

4. **overdueNotified 标志位的清理**：`BorrowRequest.overdueNotified` 一旦被设为 true，就永远不会变回 false。即使后续修改了 `expectedReturnDate`（如延期），该标志位也不会重置。这可能导致延期后再次逾期时不发通知。
